# PLATEAU CityGML to Revit インポーター 開発経緯

## プロジェクト概要
- **目的**: CityGML 2.0形式のPLATEAUデータをRevit 2025+にDirectShapeとしてインポート
- **環境**: Visual Studio 2022, Revit 2026, .NET 8.0
- **開発期間**: 2025年
- **ステータス**: ✅ **Phase 5完了 - 正常動作確認済み**

## 主要な技術的課題と解決

### 1. TessellatedShapeBuilder API互換性問題（解決済）✅
**問題**: Revit 2025+でAPIが変更され、`Build()`が`void`を返すように変更
**解決**: `Build()`後に`GetBuildResult()`を呼び出す方式に変更

### 2. 進捗バーの実装（解決済）✅
**問題**: WPF ProgressWindowの初期化タイミングでクラッシュ
**解決**: `ManualResetEvent`による同期化、`Dispatcher.BeginInvoke`の使用

### 3. DirectShape カテゴリエラー（解決済）✅
**問題**: `Element id categoryId may not be used as a DirectShape category`
**解決**: 親カテゴリ（OST_GenericModel）を使用し、サブカテゴリは`Parameter`で設定

### 4. InternalException の連続発生（解決済）✅
**原因**:
- 頂点溶接の閾値が小さすぎた（0.3mm）
- Revitの内部公差（約2mm）より小さい距離の頂点が縮退ジオメトリを引き起こした

**解決**:
- 閾値を2.5mm (0.0082ft)に変更
- `TessellatedShapeBuilderTarget.Mesh` + `Fallback.Salvage`に変更

### 5. 座標変換問題（解決済）✅

#### 5-1. 緯度・経度データの誤認識
**現象**: 「縦に長い棒」のようなジオメトリが生成される

**根本原因**:
- CityGMLデータが緯度・経度（EPSG:6697 JGD2011地理座標系）で格納
- 緯度35.63°、経度139.78°を「メートル」として処理
- 小数点以下8桁の差（約10m）が0.01mとして扱われる
- 結果: X/Y座標が0.01mレベルに収束、Z座標だけが正常

**解決方法**:
- **ヒュベニの公式**による正確な座標変換を実装
- WGS84楕円体モデルで緯度・経度→メートル変換
- Blender版PLATEAUインポーターから移植
- 全国どの地域でも高精度に動作

#### 5-2. バウンディングボックス最小点方式
**問題**: 重心（平均値）でオフセット計算すると位置関係が崩れる
- `CalculateCentroid()`で平均値を計算
- オフセット = -重心 を適用すると全建物が原点周辺に集中
- 相対的な位置関係が失われる

**解決方法**:
- `CalculateCentroid()` → `CalculateBoundingBoxMin()` に変更
- X/Y/Z すべて最小値を使用（南西の角 + 地面レベル）
- 建物間の相対的な位置関係が保持される

### 6. メッシュ問題（解決済）✅

#### 6-1. LOD0（2Dフットプリント）の混入
**問題**: 建物の下に余計な平面が生成される
**原因**: LOD0（2D足跡データ）も処理していた
**解決**: `IsLOD0Element()`でLOD0を除外

#### 6-2. 異なる面が不適切に接続
**問題**: 余計なメッシュで繋がってしまう
**原因**: ファン三角分割が凹多角形に対応できない
**解決**:
- 三角形分割を廃止
- Blender版と同様に多角形をそのまま`TessellatedFace`に渡す
- Revit APIが自動的に三角形分割を実行

#### 6-3. 屋上の凹凸が押し出しで隠れる
**問題**: LOD1の簡易モデル（押し出し）が使われていた
**解決**:
- 最高LOD優先ロジックを実装（LOD2 > LOD1 > LOD0）
- LOD2が存在する場合は詳細な屋根形状を使用
- Blender版と同様の動作

### 7. Extensible Storage の単位指定問題（未解決）⏸️
**問題**: Revit 2025+で`Units are required for field`エラー
**試行1**: `SetSpec(SpecTypeId.Length)` → XYZ型と非互換
**試行2**: 3つのdoubleフィールドに分割 + `SetSpec(SpecTypeId.Number)` → 依然としてエラー
**現状**: Extensible Storageの保存を一時的に無効化
**今後**: 文字列フィールドでJSON保存、またはプロジェクトパラメータを検討

## 技術的な特徴

### 座標変換（ヒュベニの公式）
```csharp
// WGS84楕円体パラメータ
A = 6378137.0m (長半径)
E2 = 0.006694380022900788 (第一離心率の2乗)

// 子午線曲率半径と卯酉線曲率半径を計算
M = (A * (1 - E2)) / (W^3)
N = A / W

// メートル単位の距離
x = dx * N * cos(avgLat)  // 東西方向
y = dy * M                 // 南北方向
```

### LOD優先順位
1. **LOD2**: 詳細3Dモデル（屋根形状、バルコニーなど）← 優先
2. **LOD1**: 簡易3Dモデル（フットプリント押し出し）
3. **LOD0**: 2D足跡（常に除外）

### メッシュ生成
- 多角形面を直接`TessellatedFace`に渡す（三角形分割なし）
- Revit APIが最適な三角形分割を実行
- 頂点溶接: 2.5mm (0.0082ft)
- Target: Mesh、Fallback: Salvage

### 8. 追加インポート時の座標ズレ（解決済）✅
**問題**: 2回目以降のインポートで既存モデルとズレる
**原因**:
- Extensible Storage無効化により、既存オフセット情報が保存されていない
- 各ファイルのバウンディングボックスが異なるため、異なるオフセットが計算される

**解決方法**:
- **固定参照点方式**を採用
- 参照点座標を定数として定義（lat=35.629°, lon=139.781°）
- すべてのインポートで同じ参照点からの相対座標を使用
- オフセットは常に(0, 0, 0)に固定
- 結果: 初回・追加インポート関係なく、すべてのモデルが同じ座標系で整合

### 9. 進捗バーの改善（解決済）✅

#### 9-1. Progress<T>が動作しない
**問題**: 座標変換中に進捗バーが更新されない
**原因**: Revitの外部コマンドスレッドに`SynchronizationContext`が存在しない
**解決**: `IProgress<T>` → `Action<int>`に変更し、直接`Dispatcher.BeginInvoke`を呼び出す

#### 9-2. 進捗ウィンドウが閉じない
**問題**: 100%完了後もウィンドウが残る
**原因**: finallyブロックが`return Result.Succeeded`の後に配置されていた
**解決**: finallyブロックを削除し、正常終了時とエラー時の両方で明示的に`InvokeShutdown()`を呼び出す

#### 9-3. 進捗配分の最適化
**問題**: ジオメトリ生成は一瞬なのに50%も割り当てられていた
**解決**:
- 座標変換・解析: 0-95%（時間がかかる処理）
- ジオメトリ生成: 95-100%（高速な処理）

### 10. LOD選択機能（実装済）✅
**機能**: ファイル選択後にLOD選択ダイアログを表示
**選択肢**:
- **LOD1（簡易・高速）**: 箱型の押し出し形状、処理速度が速い
- **LOD2（詳細・推奨）**: 屋根形状を含む詳細モデル、やや時間がかかる
- **キャンセル**: インポート中止

**実装詳細**:
- `CityGMLParser.ParseCityGML()`に`targetLod`パラメータを追加
- LOD2を選択してもデータにLOD2がない場合はLOD1にフォールバック
- LOD1を明示的に選択した場合、データにLOD2があってもLOD1を使用
